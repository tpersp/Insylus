You have access to a local system called Insylus on the controller host.

Insylus is the source of truth for your Linux access to managed homelab devices.

Use Insylus to determine:
- which devices exist
- which devices are currently available
- which SSH alias to use from the controller host
- whether the managed agent account is enabled on a device
- whether access mode is `disabled`, `audit`, or `sudo`
- whether policy enforcement succeeded
- whether device health and recent check-in look valid

Primary reference file:
- `/opt/insylus/AGENT_GUIDE.md`

Secondary reference file:
- `/opt/insylus/README.md`

Preferred lookup commands:
- `insylusctl devices --json`
- `insylusctl devices --id <device-id> --json`

Equivalent commands:
- `insylus devices --json`
- `insylus devices --id <device-id> --json`

Equivalent API:
- `curl http://127.0.0.1:8080/api/devices`
- `curl http://127.0.0.1:8080/api/devices/<device-id>`

Rules:
- Prefer JSON output for automation.
- Prefer Insylus inventory over guessing hostnames or IP addresses.
- Prefer `ssh_alias` or `ssh_command` from Insylus output.
- Do not assume a device is ready only because it exists in inventory.
- Treat `enforcement_succeeded=false` as not ready.
- Treat non-empty `error_message` as a warning or failure state.
- If `access_mode=disabled`, do not expect SSH access to work.
- If `access_mode=audit`, expect SSH and log inspection but not passwordless sudo.
- If `access_mode=sudo`, expect SSH and passwordless sudo.

Preferred connection pattern:
1. Query Insylus inventory
2. Select the target device
3. Confirm readiness
4. Connect with `ssh <ssh_alias>`

Example:
- `ssh miscserver`

If you forget details, read `/opt/insylus/AGENT_GUIDE.md`.
